# 📌 RAG 系统（检索增强生成）

## 1. 什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

**核心思想：** 先检索相关知识，再让 LLM 基于检索内容回答问题。

```
用户提问 → 向量化 → 检索相关文档 → 拼接 Prompt → LLM 生成答案
```

---

## 2. 为什么需要 RAG？

| 问题 | 纯 LLM | RAG |
|------|--------|-----|
| 知识过时 | ❌ 训练数据截止后不知道 | ✅ 可以检索最新数据 |
| 私有数据 | ❌ 不知道公司内部文档 | ✅ 可以检索内部知识库 |
| 幻觉 | ❌ 容易瞎编 | ✅ 基于检索内容，更准确 |
| 可追溯性 | ❌ 不知道答案从哪来 | ✅ 可以给出引用来源 |

---

## 3. RAG 全流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  文档加载器   │ ──→ │  文档切分器   │ ──→ │  Embedding   │
│  (Loader)    │     │  (Splitter)  │     │   (向量化)    │
└──────────────┘     └──────────────┘     └──────────────┘
                                                 │
                                                 ↓
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  LLM 生成答案  │ ←── │  检索器      │ ←── │  向量数据库   │
│  (Generator) │     │  (Retriever) │     │  (Vector DB) │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 两条流水线

1. **索引流水线（离线）**：文档 → 加载 → 切分 → 向量化 → 存储
2. **查询流水线（在线）**：问题 → 向量化 → 检索 → 生成答案

---

## 4. 核心组件

### 4.1 文档加载器（Loader）

支持的数据源：
- PDF、Word、Markdown、TXT
- 网页（HTML）
- 数据库
- API（Notion、Confluence 等）

### 4.2 文档切分（Chunking）

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 固定字符切分 | 按字符数硬切 | 简单快速 |
| 递归字符切分 | 按段落→句子→词语 | 保持语义完整，推荐 |
| Markdown 切分 | 按标题层级切分 | 技术文档 |
| 语义切分 | 用 embedding 聚类 | 高级玩法 |

**经验值：**
- 通用文档：500-1000 tokens
- 技术文档：256-512 tokens
- 重叠：chunk_size 的 10-20%

### 4.3 Embedding（向量化）

**主流模型：**

| 模型 | 维度 | 价格 | 中文效果 |
|------|------|------|----------|
| OpenAI text-embedding-3-small | 1536 | $0.02/1M | ⭐⭐⭐⭐ |
| OpenAI text-embedding-3-large | 3072 | $0.13/1M | ⭐⭐⭐⭐⭐ |
| BGE-M3（智源） | 1024 | 免费 | ⭐⭐⭐⭐⭐ |

### 4.4 向量数据库

| 数据库 | 类型 | 优点 | 适用场景 |
|--------|------|------|----------|
| Chroma | 嵌入式 | 轻量，开箱即用 | 开发测试 |
| Milvus | 专用 | 高性能，云原生 | 生产环境 |
| Qdrant | 专用 | Rust 编写，性能好 | 生产环境 |
| pgvector | PG 插件 | 直接用 PG | 已有 PG 基础设施 |

### 4.5 检索优化

**混合检索 = 向量检索 + 关键词检索（BM25）**

**Rerank（重排序）：**
- 先检索 50 个候选
- 用 Cross-Encoder 重新打分
- 取 top-5 送给 LLM
- 精度提升 15-25%

## 5. 引用溯源

**什么是引用溯源？**

RAG 回答时，标注答案来自哪篇文档、哪一页、哪一段。

**例子：**
```
问：报销餐费上限是多少？

答：餐费报销上限为 50 元/餐。
来源：《2024 公司报销制度.pdf》第 3 页第 2 段
```

**实现方法：**
- 给每个 Chunk 加 Metadata（文件名、页码、所属部门）
- LLM 生成答案时引用 Metadata 信息

---

## 6. RAG vs Fine-tuning

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 适用场景 | 海量、动态、非结构化知识 | 小量、固定、结构化知识 |
| 更新成本 | 低（增量更新向量库） | 高（重新训练） |
| 引用溯源 | ✅ 支持 | ❌ 不支持 |
| 幻觉控制 | 较好 | 一般 |
| 成本 | 低（按调用付费） | 高（GPU 集群） |

**选择建议：**
- 知识库问答 → RAG
- 风格迁移/格式输出 → Fine-tuning

## 📝 面试题

**Q1: RAG 的基本流程是什么？**

<details>
<summary>💡 参考答案</summary>

1. 文档加载 → 2. 切分 → 3. 向量化 → 4. 存储 → 5. 检索 → 6. 生成

</details>

**Q2: 如何解决检索结果不相关的问题？**

<details>
<summary>💡 参考答案</summary>

1. 混合检索（向量 + 关键词）
2. Multi-Query（生成多个查询变体）
3. Rerank（重新排序）
4. 优化 Embedding 模型
5. 优化 Chunking 策略

</details>

**Q3: 为什么 RAG 能提供引用溯源，而微调不能？**

<details>
<summary>💡 参考答案</summary>

- RAG 检索时保留文档元数据（文件名、页码等）
- 微调是知识内化到模型参数，无法追溯来源
- 企业场景需要可验证性，RAG 更适合

</details>
