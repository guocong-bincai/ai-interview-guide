## 一、基础概念题

### Q1: LLM 的 Token 是怎么计算的？中文和英文有什么区别？

<details>
<summary>💡 答案要点</summary>

**Token 计算规则：**

| 语言 | 计算方式 | 示例 |
|------|----------|------|
| **英文** | 1 token ≈ 4 个字符 ≈ 0.75 个单词 | "Hello world" ≈ 2-3 tokens |
| **中文** | 1 个汉字 ≈ 1-2 个 tokens | "你好世界" ≈ 4-6 tokens |

**影响因素：**
1. **分词算法**：BPE、WordPiece、Unigram 等
2. **词表大小**：32k-100k tokens
3. **特殊字符**：空格、标点、emoji 单独计算

**实际影响：**
- **计费**：输入 + 输出都按 token 计费
- **上下文限制**：128K 指的是 tokens，不是字符数
- **生成长度**：max_tokens 限制输出长度

**面试话术：**
> "中文的 token 效率比英文低，同样长度的文本，中文 token 数可能是英文的 2-3 倍。我在项目中会监控 token 消耗，对于高频场景，用 Prompt 压缩和缓存来降低成本。"

</details>

### Q2: 什么是 Context Window？超出限制怎么处理？

<details>
<summary>💡 答案要点</summary>

**常见模型的上下文限制：**

| 模型 | Context Window | 适用场景 |
|------|----------------|----------|
| GPT-4o | 128K tokens | 长文档分析 |
| GPT-3.5 | 16K tokens | 常规对话 |
| Claude 3 | 200K tokens | 超长文本 |
| Llama 3 | 8K-32K tokens | 开源部署 |

**超出限制的解决方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **截断** | 只保留最近的对话 | 简单，但丢失历史 |
| **滑动窗口** | 保留最近 N 轮 | 平衡，但可能丢失关键信息 |
| **摘要压缩** | 用 LLM 总结历史对话 | 保留核心，但有信息损失 |
| **向量检索** | 把历史存向量库，按需检索 | 最灵活，但实现复杂 |
| **分层管理** | 重要信息摘要 + 最近对话原文 | 最佳实践 |

**面试话术：**
> "我用分层策略：最近 3 轮保留原文，更早的对话用 LLM 总结。同时加入向量检索，如果用户提到之前的内容，可以从向量库检索相关历史。这样既控制了成本，又保证了对话连贯性。"

</details>

### Q3: Temperature、Top-P、Top-K 怎么调？有什么影响？

<details>
<summary>💡 答案要点</summary>

**参数说明：**

| 参数 | 作用 | 取值范围 | 影响 |
|------|------|----------|------|
| **Temperature** | 控制随机性 | 0-2 | 越低越确定，越高越随机 |
| **Top-P** | 核采样阈值 | 0-1 | 只从累积概率>P 的词里选 |
| **Top-K** | 候选词数量 | 1-100 | 只从概率最高的 K 个词里选 |

**推荐配置：**

| 场景 | Temperature | Top-P | Top-K |
|------|-------------|-------|-------|
| **RAG/问答** | 0-0.3 | 0.9 | - |
| **创意写作** | 0.7-1.0 | 0.9 | - |
| **代码生成** | 0.2-0.3 | 0.95 | 50 |
| **数据分析** | 0-0.1 | 0.9 | - |
| **聊天对话** | 0.7-0.9 | 0.9 | - |

**面试话术：**
> "Temperature 是最重要的参数。问答场景我用 0-0.3，保证答案稳定；创意场景用 0.7-1.0，让输出更多样。Top-P 和 Top-K 是辅助控制，一般用默认值就够了。"

</details>

## 二、架构设计题

### Q4: 如何设计一个支持 10 万并发用户的 AI 问答系统？

<details>
<summary>💡 答案要点</summary>

**架构设计：**
```
用户 → CDN → 负载均衡 → API Gateway → 服务集群 → LLM API
                                    ↓
                              Redis 缓存
                                    ↓
                              向量数据库
                                    ↓
                              监控告警
```

**关键技术点：**

| 层级 | 方案 | 说明 |
|------|------|------|
| **接入层** | CDN + 负载均衡 | 分散流量，降低延迟 |
| **网关层** | 限流 + 鉴权 | 令牌桶限流，防止雪崩 |
| **服务层** | 多实例 + 自动扩缩容 | K8s HPA，根据 CPU/内存自动扩容 |
| **缓存层** | Redis + 语义缓存 | 热数据缓存，相似问题直接返回 |
| **数据库** | 向量库 + 关系库 | 分离读写，提升性能 |
| **监控** | Prometheus + Grafana | 实时监控，自动告警 |

**并发优化：**
1. **异步处理**：用消息队列削峰填谷
2. **流式输出**：降低首字延迟，提升用户体验
3. **多 Key 轮询**：突破 LLM API 的 RPM 限制
4. **降级策略**：LLM 挂了返回预设答案

**面试话术：**
> "我设计过一个支撑 200 QPS 的系统，核心是三层缓存：精确缓存（相同问题）、语义缓存（相似问题）、预设缓存（FAQ）。同时用多 Key 轮询突破 OpenAI 的 RPM 限制，用流式输出提升用户体验。最终卡死率从 15% 降到 0.5%。"

</details>

<details>
<summary>💡 答案要点</summary>

**核心挑战：**
1. 数据隔离（不同租户的数据不能混）
2. 权限控制（不同用户访问不同知识）
3. 资源配额（防止某个租户占用所有资源）

**架构设计：**
```
┌─────────────────────────────────────────────────────────┐
│                   多租户 AI 知识库系统                    │
└─────────────────────────────────────────────────────────┘

用户 → 鉴权 → 租户识别 → 权限过滤 → 检索 → 生成 → 返回
                    ↓
            ┌───────┼───────┐
            ↓       ↓       ↓
        租户 A    租户 B    租户 C
        (隔离)   (隔离)   (隔离)
```

**实现方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **物理隔离** | 每个租户独立的向量库 | 安全，但成本高 |
| **逻辑隔离** | 同一个向量库，用 metadata 过滤 | 成本低，但需要严格权限控制 |
| **混合方案** | 大租户独立，小租户共享 | 平衡 |

**权限控制：**
```python
# 检索时加入租户和权限过滤
results = vectorstore.search(
    query=query,
    filter={
        "tenant_id": tenant_id,
        "department": {"$in": user_departments},
        "access_level": {"$lte": user_level}
    }
)
```

**面试话术：**
> "我用逻辑隔离方案，在向量库的 metadata 里加入 tenant_id、department、access_level 等字段。检索时自动过滤，确保用户只能访问授权的知识。同时做了资源配额，每个租户有独立的 Token 预算，防止滥用。"

### Q9: 如何处理 AI 应用的并发限流问题？

<details>
<summary>💡 答案要点</summary>

**限流策略：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **令牌桶** | 固定速率产生令牌，用完等待 | 常规限流 |
| **漏桶** | 固定速率流出，超出丢弃 | 平滑流量 |
| **滑动窗口** | 统计最近 N 秒请求数 | 精确控制 |
| **分布式限流** | Redis + Lua 脚本 | 多实例场景 |

**OpenAI API 限制：**
- RPM（Requests Per Minute）：每分钟请求数
- TPM（Tokens Per Minute）：每分钟 token 数
- 解决方案：多 Key 轮询

**实现示例（Go）：**
```go
type RateLimiter struct {
    tokens chan struct{}
}

func NewRateLimiter(rate int) *RateLimiter {
    rl := &RateLimiter{tokens: make(chan struct{}, rate)}
    go func() {
        for {
            time.Sleep(time.Minute / time.Duration(rate))
            rl.tokens <- struct{}{}
        }
    }()
    return rl
}

func (rl *RateLimiter) Wait() {
    <-rl.tokens
}
```

**面试话术：**
> "我在 API Gateway 层实现了令牌桶限流，防止瞬时请求压垮后端。针对 OpenAI 的 RPM 限制，我设计了多 Key 轮询机制，5 个 Key 轮流使用，相当于把限流阈值提高了 5 倍。同时加入了请求队列，超出的请求排队等待，而不是直接拒绝。"

</details>

### Q11: 如何处理用户隐私数据？（PII 脱敏、数据留存）

<details>
<summary>💡 答案要点</summary>

**隐私数据类型：**
- 个人身份信息（PII）：姓名、身份证、手机号
- 敏感信息：地址、邮箱、银行卡
- 企业机密：合同、财务数据

**脱敏方案：**

| 数据类型 | 脱敏方式 | 示例 |
|----------|----------|------|
| **手机号** | 中间 4 位掩码 | 138****1234 |
| **身份证** | 保留前 3 后 4 | 110***********1234 |
| **邮箱** | 保留首尾 | g***g@163.com |
| **姓名** | 只保留姓 | 张** |

**实现示例：**
```python
import re

def sanitize_pii(text):
    # 手机号脱敏
    text = re.sub(r'1[3-9]\d{9}', lambda m: m.group()[:3] + '****' + m.group()[-4:], text)
    # 身份证脱敏
    text = re.sub(r'\d{18}', lambda m: m.group()[:3] + '***********' + m.group()[-4:], text)
    # 邮箱脱敏
    text = re.sub(r'(\w)[\w.]*(@\w+\.\w+)', lambda m: m.group(1) + '***' + m.group(2), text)
    return text
```

**数据留存策略：**
- **日志**：脱敏后留存 30 天
- **对话历史**：用户可删除，系统定期清理
- **模型训练**：需用户明确授权

**面试话术：**
> "我在输入和输出层都做了 PII 脱敏，用正则匹配手机号、身份证、邮箱等敏感数据。日志脱敏后留存 30 天，对话历史用户可以随时删除。另外，所有用于模型训练的数据都获得了用户明确授权，符合 GDPR 要求。"

</details>

| **安全检测** | 静态分析 | 检测潜在的安全漏洞 |

### Q14: 设计一个 AI 数据分析系统，让非技术人员也能分析数据

<details>
<summary>💡 答案要点</summary>

**核心功能：**
1. 自然语言查询（"上个月销售额是多少？"）

3. 数据解读（"分析销售下降的原因"）

**架构设计：**
```
用户问题 → NL2SQL → SQL 校验 → 执行 → 结果 → 可视化 → 解读
              ↓                                          ↓
         表结构注入                                   LLM 生成
```

**关键技术点：**

| 模块 | 方案 | 说明 |

| **NL2SQL** | Prompt + 表结构 + Few-shot | 生成准确 SQL |

| **可视化** | ECharts / Chart.js | 自动选择图表类型 |
| **数据解读** | LLM + 业务规则 | 生成洞察报告 |

**面试话术：**
> "我设计的数据分析系统，核心是 NL2SQL 和可视化自动化。NL2SQL 加入了三层防护：语法检查、安全过滤、执行限流，防止危险 SQL。可视化根据数据类型自动选择图表（趋势用折线图、对比用柱状图）。最后用 LLM 生成数据解读，让非技术人员也能理解数据背后的业务含义。"
