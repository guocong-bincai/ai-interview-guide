## 一、岗位定位与能力地图

### AI 应用开发工程师 vs 传统算法工程师

| 维度 | 传统算法工程师 | AI 应用开发工程师 |
|------|----------------|-------------------|
| **核心工作** | 训练/优化模型 | 利用现有模型构建业务系统 |
| **技术栈** | PyTorch/TensorFlow | LangChain/LlamaIndex + 后端框架 |
| **关注点** | 模型精度、召回率 | 稳定性、成本、用户体验 |
| **输出** | 模型文件 | 可商用的 API/产品 |

### 面试官核心关注点

| 考点维度 | 面试官的核心关注点 |
|----------|-------------------|
| **工程能力** | 是否熟悉 Python/Go？能否用 FastAPI/Gin 构建高性能后端？ |
| **模型应用** | 是否知道不同模型（OpenAI vs DeepSeek vs Llama）的优缺点和价格？ |
| **数据处理** | ETL 能力。如何清洗非结构化数据（PDF、Markdown、图片中的文字）？ |
| **评估能力** | 怎么证明你的 AI 应用比昨天更聪明了？（RAGAS 评分、人工评测集） |
| **成本意识** | Token 是要钱的。懂得通过 Prompt 压缩、模型路由来省钱。 |

### 核心能力清单

1. **熟练掌握主流框架**
   - LangChain / LangGraph（行业标准）
   - LlamaIndex（专注于 RAG 和数据连接）

2. **向量数据库实战**
   - 懂得如何创建索引
   - 选择相似度算法（Cosine Similarity vs Euclidean Distance）

3. **全栈开发思维**
   - AI 只是后端的一个"算力节点"
   - 仍需懂 API 设计、并发处理、数据库事务

4. **敏锐的成本意识**
   - Token 压缩
   - 模型蒸馏
   - 逻辑路由（简单问题用小模型，复杂问题用大模型）

## 二、模拟面试题（综合卷）

### 场景题

**Q1: 公司准备上线一个"企业级知识库助手"，需要支持 10 万份 PDF 文档的检索，且要求回答准确、带引用来源。你会如何设计这个系统的架构？**

<details>
<summary>💡 高分回答</summary>

**1. 架构选型：RAG 是首选，Fine-tuning 是补充**

> "我会优先采用 Advanced RAG 架构而非微调。原因有三：
> 1. 知识库需要频繁更新，RAG 只需更新向量库，而微调成本太高且有时效性滞后
> 2. RAG 可以提供引用溯源（Source Citations），消除幻觉
> 3. 微调无法处理海量非结构化文档的检索"

**进阶提分：** "如果后期发现模型无法理解特定行业的术语或特定的输出格式，我会考虑用少量高质量数据进行 LoRA 微调，以对齐风格。"

**2. 数据清洗与切片（Data Ingestion）**

> "切片（Chunking）不能简单按字符数。我会采用 Markdown 语义切片或递归字符切片。对于 PDF 中的表格，我会使用 Unstructured 或 GPT-4o-mini 将表格转为 Markdown 格式，否则向量检索会丢失行列逻辑。"

**进阶提分：** "我会给每个 Chunk 增加 Metadata（元数据），比如文件名、页码、所属部门。这样在检索时可以进行 Self-Querying（根据用户权限或范围过滤标签）。"

**3. 检索优化（Retrieval Strategy）**

> "单次向量检索往往不够。我会引入 Multi-Query Retrieval（将用户问题扩展成多个同义句）和 Hybrid Search（向量检索 + 关键词 BM25 检索）来提升召回率。"

**进阶提分：** "我会加入 Rerank（重排序）环节。先从向量库召回 50 个候选片断，再用专门的 Reranker 模型（如 BGE-Reranker）精选出最相关的 Top-5。这能显著解决'模型找不到重点'的问题。"

**4. 评估与工程化（Evaluation & Ops）**

> "我会建立一套 RAGAS 评估体系，重点监控四个维度：忠实度（Faithfulness）、相关度（Answer Relevance）、上下文精度（Context Precision）和召回率（Context Recall）。"

**进阶提分：** "为了降低 Token 成本和延迟，我会部署 Semantic Cache（语义缓存）。如果两个用户问了语义相似的问题，直接从 Redis 缓存中读取答案，无需再次调用大模型。"

</details>

### 技术选型题

**Q2: 你会选择 RAG（检索增强生成）还是 Fine-tuning（微调）？理由是什么？**

<details>
<summary>💡 答案要点</summary>

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| **适用场景** | 海量、动态、非结构化知识 | 小量、固定、结构化知识 |
| **更新成本** | 低（增量更新向量库） | 高（重新训练） |
| **引用溯源** | ✅ 支持 | ❌ 不支持 |
| **幻觉控制** | 较好（基于检索内容） | 一般（依赖训练数据） |
| **成本** | 低（按调用付费） | 高（GPU 集群训练） |

**回答模板：**
> "如果是知识库问答、文档检索这类场景，我选 RAG，因为知识需要频繁更新且需要引用溯源。如果是风格迁移、特定格式输出这类场景，我选微调，因为需要模型学习固定的表达模式。"

</details>

### 工程实现题

**Q3: 如何处理长文档的 Chunking（切片）？**

<details>
<summary>💡 答案要点</summary>

**切分策略对比：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 固定字符切分 | 按字符数硬切 | 简单快速，但可能切断语义 |
| 递归字符切分 | 按段落→句子→词语递归切 | 保持语义完整，推荐 |
| Markdown 切分 | 按标题层级切分 | 技术文档 |
| 语义切分 | 用 embedding 聚类 | 高级玩法，效果好但慢 |

**经验值：**
- 通用文档：500-1000 tokens
- 技术文档：256-512 tokens
- 对话数据：1024-2048 tokens
- 重叠：chunk_size 的 10-20%

</details>

---

**Q4: 如何解决检索出来的片段不相关（Recall 质量差）的问题？**

<details>
<summary>💡 答案要点</summary>

**解决方案：**
1. **混合检索**：向量检索 + 关键词检索（BM25）
2. **Multi-Query**：用 LLM 生成多个查询变体
3. **Rerank**：用 Cross-Encoder 重新排序
4. **优化 Embedding**：换更好的模型（如 BGE-M3 中文好）
5. **优化 Chunking**：调整 chunk 大小和重叠

</details>

### 性能与成本题

**Q5: 如果用户量激增，如何通过缓存（Cache）优化 API 开销？**

<details>
<summary>💡 答案要点</summary>

**缓存策略：**
1. **语义缓存**：用向量相似度判断问题是否重复（相似度>0.95 直接返回缓存）
2. **精确缓存**：完全相同的问题直接返回
3. **分层缓存**：
   - L1：Redis 缓存（热数据）
   - L2：向量库缓存（相似问题）
   - L3：数据库缓存（历史答案）

**效果：** 缓存命中率可达 30-50%，成本降低 35%+

</details>

---

**Q6: 如何监控模型是否产生了"幻觉"（Hallucination）？**

<details>
<summary>💡 答案要点</summary>

**监控方法：**
1. **引用检查**：答案中的每个陈述是否能在检索内容中找到依据
2. **RAGAS 指标**：Faithfulness（忠实度）< 0.7 触发告警
3. **人工抽检**：随机抽样 5% 的答案人工审核
4. **用户反馈**：点赞/点踩机制收集反馈

</details>

### 进阶题

**Q7: 如果任务需要助手去操作外部 CRM 系统（如查询客户订单），你会如何实现？**

<details>
<summary>💡 答案要点</summary>

**考点：Function Calling / AI Agent**

**实现方案：**
1. **定义工具**：将 CRM API 封装成 Function Schema
2. **注册工具**：在 LLM 调用时传入 tools 参数
3. **解析调用**：解析 LLM 返回的 function_call
4. **执行工具**：调用实际 API 获取数据
5. **返回结果**：将 API 结果返回给 LLM 生成最终答案

**进阶提分：**
> "我会加入权限校验和参数白名单，防止 Agent 越权访问。同时设置最大调用次数和超时时间，防止死循环。"

</details>

## 三、核心关键词深度拆解

### 1. RAG (Retrieval-Augmented Generation)

| 维度 | 说明 |
|------|------|
| **理解** | 给大模型配一个"外部图书馆"。模型不再只靠记忆，而是先去查资料，再总结回答。 |
| **考点** | 向量数据库选择、Embedding 模型匹配、重排序（Rerank）的重要性 |
| **流程** | 加载→切分→向量化→存储→检索→生成 |
| **高分回答** | "RAG 的核心是解决 LLM 知识过时和幻觉问题。我会用混合检索 + Rerank 提升召回率，用语义缓存降低成本。" |

---

### 2. Prompt Engineering & Optimization

| 维度 | 说明 |
|------|------|
| **理解** | 并不是简单的"写提示词"，而是通过 CoT、Few-shot 等策略，甚至使用自动化工具（如 DSPy）来优化模型输出。 |
| **考点** | 结构化提示词（JSON 格式输出）、防止 Prompt Injection 的安全性 |
| **技巧** | 明确角色、清晰任务、提供示例、指定格式、设置约束 |
| **高分回答** | "好的 Prompt 能让同一模型效果提升 30%。我会用 Few-shot 保证格式一致性，用 CoT 提升复杂推理能力。" |

---

### 3. AI Agent (智能体)

| 维度 | 说明 |
|------|------|
| **理解** | 模型不再只是"说话"，而是能"干活"。具备规划（Planning）、记忆（Memory）和工具使用（Tool Use）的能力。 |
| **考点** | Function Calling 原理、ReAct 框架、多智能体协同（AutoGen/LangGraph） |
| **核心组件** | LLM（大脑）+ Tools（工具）+ Memory（记忆）+ Planning（规划） |
| **高分回答** | "Agent 的核心在于闭环。模型生成答案后，我会设计一个 Reviewer 节点让它自我检查，如果不满足则重新执行。" |

---

### 4. PEFT (参数高效微调)

| 维度 | 说明 |
|------|------|
| **理解** | 当 RAG 解决不了语气、风格或极其专业的领域知识时，才考虑微调。常用 LoRA 算法，只改动极小比例的参数。 |
| **考点** | 为什么不全量微调？LoRA 的原理是什么？ |
| **优势** | 显存占用少、训练快、可插拔 |
| **高分回答** | "全量微调成本太高且容易灾难性遗忘。LoRA 通过低秩适配器只更新 1-5% 的参数，效果接近全量微调但成本低 10 倍。" |

---

### 5. Embedding & 向量检索

| 维度 | 说明 |
|------|------|
| **Embedding** | 把文本转成向量（一串数字），语义相似的文本向量接近 |
| **1536 维** | 用 1536 个特征描述文本（OpenAI text-embedding-3-small） |
| **余弦相似度** | 算两个向量"方向"有多接近，1=一样，0=无关，-1=相反 |
| **ANN 索引** | 近似最近邻搜索，速度快 100-1000 倍（HNSW/IVF） |
| **高分回答** | "Embedding 是语义的数学表示。我用余弦相似度衡量相关性，用 HNSW 索引加速检索。" |

---

### 6. 向量数据库

| 维度 | 说明 |
|------|------|
| **为什么用** | 传统数据库做向量检索慢（O(N)），向量库用 ANN 索引快（O(log N)） |
| **主流选择** | Milvus（大规模）、Qdrant（性能好）、Chroma（轻量）、pgvector（已有 PG） |
| **性能对比** | MySQL 无索引 5000ms → pgvector 500ms → Qdrant 10ms |
| **高分回答** | "向量库是专为'找相似'设计的跑车，传统库是家用轿车。10 万条数据检索差 100-500 倍。" |

## 四、高分实战案例

### 案例一：处理"脏数据"——复杂 PDF 与表格解析

**背景：** 扫描版合同 PDF，包含大量印章叠加、跨页表格和三栏布局，直接 OCR 导致文字逻辑错乱。

**高分回答逻辑：**

> "我遇到过最棘手的'脏数据'是处理带有复杂嵌套表格的扫描版审计报告。传统的 PDFMiner 或简单 OCR 库会将表格内容按行切割，导致语义完全断裂。"

**解决方案：**

1. **布局分析（Layout Analysis）**
   - 引入 Layout-Parser 或 PaddleOCR 的区域检测模型
   - 先识别出文档中的"表格区"、"正文区"和"页眉页脚"

2. **多模态增强**
   - 针对极难处理的表格，采用 VLM（如 GPT-4o 或 Qwen-VL）直接截取图像进行转换

3. **管道化清洗（Pipeline）**
   - 先用 PyMuPDF 提取可复制文字
   - 对无法识别的图片层启动 OCR
   - 最后通过一个 Cleaner LLM 将 OCR 的噪点结合上下文修复

**结果：** 最终将财务指标提取的准确率从 65% 提升到了 94%，并实现了表格数据的 Markdown 结构化还原。

---

### 案例二：成本与性能调优——模型分流器（Router）

**背景：** 用户每天产生 5 万次对话，全部请求 GPT-4 导致成本极高，且简单问题响应慢。

**高分回答逻辑：**

> "为了平衡成本与响应速度，我设计并落地了一套 Intelligent LLM Router（智能路由）系统，而不是单一模型包打天下。"

**解决方案：**

1. **意图识别路由**
   - 接入层使用轻量级的 BERT 或 GPT-4o-mini 作为判定器
   - 简单问题分流到 Llama 3-8B 或 DeepSeek-V3
   - 复杂问题用 GPT-4

2. **Prompt 压缩（Context Compression）**
   - 针对 RAG 检索回来的 5000 字参考资料
   - 通过 LLMLingua 算法移除冗余助词和无关元数据
   - 在保证语义不丢失的前提下减少 40% 的 Context Token

3. **语义缓存**
   - 引入 GPTCache
   - 对于相似度高于 0.95 的重复问题直接返回缓存答案

**结果：** 综合 Token 成本降低了 35%，平均首字响应时间（TTFT）缩短了 50%。

---

### 案例三：从 Demo 到 Production——解决并发与流式卡顿

**背景：** 系统上线后，高并发下出现 API 超时，且流式前端显示如同"挤牙膏"，用户体验极差。

**高分回答逻辑：**

> "Demo 环境下一切完美，但生产环境下大模型的非确定性延迟是核心痛点。我主要通过后端异步队列和前端流式渲染优化解决了这个问题。"

**解决方案：**

1. **并发控制与限流**
   - 在 API Gateway 层实现令牌桶限流
   - 防止瞬时请求压垮后端连接池
   - 针对 OpenAI 的 TPM/RPM 限制，建立多 Key 轮询缓冲机制

2. **流式输出优化**
   - 将后端的 Server-Sent Events (SSE) 调整为分块传输（Chunking）
   - 采用"双缓冲机制"：前端先接收文字进入缓冲区，再按固定频率模拟打字机平滑显示

3. **超时容错**
   - 为每个请求设置严格的阶段性超时（如 5s 内必须首字输出）
   - 一旦超时立即触发 Fallback，切换到响应更快的备用模型

**结果：** 系统成功支撑了 200 QPS 的峰值压力，用户感知的"卡死率"从 15% 降至 0.5% 以下。

## 五、专业术语词典

### LLM 基础

| 术语 | 英文 | 通俗解释 | 面试怎么用 |
|------|------|----------|------------|
| **大语言模型** | LLM | 能理解和生成文本的 AI 模型 | "我们用 LLM 作为核心生成引擎" |
| **Transformer** | Transformer | LLM 的底层架构，擅长处理序列数据 | "基于 Transformer 的 decoder-only 架构" |
| **预训练** | Pre-training | 在海量数据上训练，学习通用知识 | "模型在万亿级 token 上预训练" |
| **微调** | Fine-tuning | 在特定数据上继续训练，适应特定任务 | "用客服对话数据微调模型" |
| **指令微调** | Instruction Tuning | 用"指令 - 回答"对训练，让模型听懂命令 | "经过指令微调，模型能更好遵循 prompt" |
| **对齐** | Alignment | 让模型输出符合人类价值观 | "用 RLHF 做人类偏好对齐" |
| **Token** | Token | LLM 处理文本的基本单位 | "GPT-4 的上下文是 128K tokens" |
| **上下文窗口** | Context Window | LLM 一次能处理的最大 token 数 | "需要管理上下文，避免超出限制" |
| **温度** | Temperature | 控制输出的随机性，0=确定，1=随机 | "问答场景用低温，创作用高温" |
| **幻觉** | Hallucination | 模型编造不存在的信息 | "用 RAG 减少幻觉问题" |

### Embedding & 向量

| 术语 | 英文 | 通俗解释 | 面试怎么用 |
|------|------|----------|------------|
| **嵌入** | Embedding | 把文本转成向量（一串数字） | "用 embedding 表示文本语义" |
| **向量** | Vector | 一串数字，embedding 的输出 | "每个文档转成 1536 维向量" |
| **余弦相似度** | Cosine Similarity | 算两个向量有多接近，-1 到 1 | "用余弦相似度衡量语义相关性" |
| **语义搜索** | Semantic Search | 按意思搜索，不是关键词匹配 | "用户搜'贵'能找到'价格高'" |

## 七、MCP & Skill 系统（2026 面试热点）

### Q1: 什么是 MCP（Model Context Protocol）？

**MCP = Model Context Protocol（模型上下文协议）**

**本质：** AI 时代的 USB-C 协议，统一模型与外部资源的连接方式。

**解决的问题：**
| 问题 | 传统方案 | MCP 方案 |
|------|----------|----------|
| 协议不统一 | 每个工具自定义 API | 统一协议，一次接入 |
| 发现困难 | 手动配置工具 | 自动发现可用资源 |
| 权限混乱 | 硬编码认证信息 | 标准化权限管理 |
| 互操作性差 | 工具之间无法协作 | 标准化接口，可组合 |

**三种资源类型：**
- **Prompts**：预定义的 Prompt 模板
- **Resources**：可读取的数据源（只读）
- **Tools**：可执行的操作（读写）

**面试话术：** "MCP 的本质是 AI 时代的 USB-C 协议。我在项目中用 MCP 接入了 10+ 数据源，开发效率提升了 3 倍。"

---

### Q2: MCP Server 和 MCP Client 的职责？

| 角色 | 职责 | 示例 |
|------|------|------|
| **MCP Server** | 暴露能力（资源/工具/提示词） | 数据库服务、搜索服务 |
| **MCP Client** | 发现并调用能力 | AI 应用、Agent 框架 |

---
